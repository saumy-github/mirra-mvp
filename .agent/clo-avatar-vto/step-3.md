# Step 3 — CLO Native VTO (Virtual Try-On)

**Command:** `python clo_avatar_generation/run_clo_vto.py`

This is the Step 3 implementation of the Mirra MVP. It takes the personalized avatar produced by Step 1 and a set of garment pattern files (DXF) produced by the product ingestion pipeline, then assembles and simulates the garment on the avatar inside CLO via the REST plugin.

---

## What It Does

1. Opens a fresh CLO project
2. Imports the user's personalized avatar (output of Step 1)
3. Imports DXF garment pattern pieces (output of product ingestion)
4. Verifies pattern count and scale
5. Reads edge data from each pattern piece
6. Matches each piece to an arrangement slot on the avatar body
7. Arranges pattern pieces around the avatar in 3D
8. Applies fabric material to pieces
9. Creates seams between pieces (joins front/back/sleeves)
10. Runs physics simulation (150 steps)
11. Prints export note (export/save is currently disabled in code)

---

## How to Run

### Basic (auto-discovers inputs)
```
python clo_avatar_generation/run_clo_vto.py
```

Auto-discovers:
- Avatar: looks for `output/clo_test.avt` → any `.avt` in `output/` → `avt_templates/` → `input/base-1.avt`
- Patterns: uses `product_ingestion.run_manifest.get_latest_panels_dxf_dir()` to find latest DXF output

### With explicit paths
```
python clo_avatar_generation/run_clo_vto.py \
  --avt-path clo_avatar_generation/output/u_001-001/result_avatar.avt \
  --patterns-dir product_ingestion/output/panels/dxf \
  --report-path clo_avatar_generation/output/vto_report.json
```

### With measurement CSV (optional, not reliable)
```
python clo_avatar_generation/run_clo_vto.py --use-default-csv
```

### All CLI flags

## Inputs

### 1. Avatar `.avt` file
The personalized avatar produced by Step 1.
Discovery order:
1. `--avt-path` flag
2. `output/clo_test.avt`
3. Any single `.avt` in `output/`
4. Any single `.avt` in `avt_templates/`
5. `input/base-1.avt` (fallback to base)

### 2. DXF Pattern files
Located in `patterns_dir/` — must contain these 4 files:
```
front_panel.dxf
back_panel.dxf
sleeve_left.dxf
sleeve_right.dxf
```
Source: output of `product_ingestion` pipeline.
Auto-resolved via `product_ingestion.run_manifest.get_latest_panels_dxf_dir()`.

### 3. Seam map
Three resolution layers (in priority order):
1. Explicit `seam_map` argument passed to `run_pipeline()`
2. `edge_manifest.json` found alongside the DXF files (auto-generated by product ingestion)
3. `DEFAULT_SEAMS` — hardcoded 10-seam fallback (placeholder edge indices)

### 4. CLO REST Plugin
Must be running at `http://localhost:50505`.

## The 11-Step Pipeline

| Step | File | Description |
|---|---|---|
| 01 | `step_01_health.py` | Health-check CLO plugin |
| 02 | `step_02_new_project.py` | Create fresh CLO project |
| 03 | `step_03_import_avatar.py` | Import personalized avatar `.avt` |
| 04 | `step_04_import_patterns.py` | Import 4 DXF pattern pieces, validate scale |
| 05 | `step_05_verify_patterns.py` | Verify correct pattern count was loaded |
| 06 | `step_06_read_edges_and_slots.py` | Read edge counts per piece, resolve arrangement slots |
| 07 | `step_07_arrange_patterns.py` | Place pattern pieces around avatar in 3D |
| 08 | `step_08_apply_fabric.py` | Apply fabric material to pattern pieces |
| 09 | `step_09_create_seams.py` | Stitch pieces together with seam lines |
| 10 | `step_10_simulate.py` | Run CLO physics simulation (150 steps) |
| 11 | `step_11_export_note.py` | Print export instructions (export disabled in code) |

Pipeline stops at the first failed step (no `always_run` steps).

## Key Implementation Details

### Pattern import & scale detection (Step 4)
- Uses `ezdxf` to read DXF geometry and detect the unit system
- If DXF is in millimeters → import scale is set to `0.1` (mm → CLO cm)
- Otherwise → scale `1.0`
- Validates bounding box: each piece must be 5–300 cm (or 50–3000 mm)
- Computes SHA-256 hash of each DXF file for geometry fingerprinting

### Slot matching (Step 6)
After importing patterns, the pipeline queries CLO for arrangement slots on the avatar body and matches each pattern piece to the right body region.

Slot matching priority (per piece):
1. Exact name match against a preferred slot name list (e.g. `Body_Front_Center_2`)
2. Name fragment match (e.g. slots containing `"arm"` + `"_l"`)
3. Keyword scoring (e.g. `["body", "front"]` with optional `["torso", "chest"]`)

Required slots: `front`, `back`, `sleeve_L`, `sleeve_R`

If CLO doesn't expose slots (some runtime versions): falls back to `pattern-arrangements-no-slots` mode where patterns are placed using hardcoded offsets.

**Slot warmup:** if the arrangement list is empty on first query, the pipeline retries up to 3 times with a brief simulation step between each attempt.

### Seam creation (Step 9)
The 10-seam map connects the 4 pieces:

| Seam | Pieces |
|---|---|
| shoulder-right / shoulder-left | front ↔ back |
| side-right / side-left | front ↔ back |
| sleeve-L-tube | left sleeve underarm (self-seam) |
| sleeve-R-tube | right sleeve underarm (self-seam) |
| arm-R-front / arm-R-back | front/back ↔ right sleeve cap |
| arm-L-front / arm-L-back | front/back ↔ left sleeve cap (cap orientation reversed) |

Each seam entry specifies: `piece_a`, `edge_index_a`, `piece_b`, `edge_index_b`, direction flags.

**Seam geometry hash check:** before creating seams, the pipeline compares the geometry hash of the imported DXF files against the hash stored in `DEFAULT_SEAM_META`. If they don't match, sewing is aborted. Currently `DEFAULT_SEAM_META.geometry_hash` is empty (`""`), so this check is bypassed at runtime.

### Physics simulation (Step 10)
- Runs 150 simulation steps in CLO
- Waits up to 300 seconds for the queue to drain
- Skipped if no avatar is loaded (to prevent CLO crash)

### Export (Step 11)
**Export is currently disabled.** Step 11 just prints:
```
Export + Save skipped (re-enable in code when ready)
To export manually: File -> Export -> glTF 2.0
To save manually:   File -> Save As -> .zprj
```

After the pipeline finishes, the CLO project is left open — the user must export manually.

## Outputs

| Output | Description |
|---|---|
| `output/{avt_stem}__native_vto_report.json` | Full pipeline report with step results and all diagnostics |

- Pattern piece → CLO index mapping

- Scale metrics, pattern geometry hash

## Current Status

### What is working
- Health check and project creation
- Avatar import from the Step 1 `.avt` output
- DXF pattern import with automatic scale detection (mm → cm)
- Edge count reading (SDK probe → DXF fallback)
- Slot resolution with multi-tier fallback logic
- Pattern arrangement (both slot-based and degraded offset mode)
- Fabric material application
- Seam creation from the seam map
- Physics simulation (150 steps)

### Known issues / limitations

**Export not automated:**
Step 11 is a no-op. After the pipeline runs, the result must be manually exported from CLO UI (`File → Export → glTF 2.0`). Re-enabling this requires either a working CLO export plugin endpoint or a save+extract approach like Step 1 uses.

**Seam edge indices are placeholders:**
`DEFAULT_SEAMS` uses hardcoded edge indices (0–7 for body panels, 0–4 for sleeves). These are only correct if the DXF pattern files were generated with the exact same edge ordering. If the DXF geometry changes, seam indices must be rediscovered using `plugins/discover_seam_indices.py`.

**Seam geometry hash is not set:**
`DEFAULT_SEAM_META.geometry_hash` is an empty string, so the geometry compatibility check that guards seam creation is bypassed. This means seam creation proceeds even if the DXF files have changed since the seam map was last verified.

**Slot matching can fail:**
On some CLO runtime versions, arrangement slots are not populated until after a simulate step. The warmup retry handles this, but if the base avatar has an unusual slot layout, manual slot name mapping may be needed.

**4-piece T-shirt only:**
The VTO pipeline is hardcoded for exactly 4 pieces: `front_panel`, `back_panel`, `sleeve_left`, `sleeve_right`. Other garment types would require changes to the pattern files list, seam map, and slot resolution logic.

**Measurement CSV ignored:**
The `--csv-path` / `--use-default-csv` flags are wired in but the measurement CSV is not actually used in the VTO simulation. Measurements come from the avatar itself (already baked in during Step 1).

## Connection to Step 1

The `.avt` file produced by Step 1 is the direct input to Step 3:

```
Step 1 output:
  clo_avatar_generation/output/u_001-001/result_avatar.avt

Step 3 input (via --avt-path or auto-discovery):
  clo_avatar_generation/output/u_001-001/result_avatar.avt
```

The avatar already has the user's measurements baked in at the binary level. Step 3 does not re-apply measurements — it uses the avatar as-is and drapes the garment over it.

## Connection to Product Ingestion

The DXF pattern files come from the `product_ingestion` pipeline:
```
product_ingestion/output/panels/dxf/
  front_panel.dxf
  back_panel.dxf
  sleeve_left.dxf
  sleeve_right.dxf
  ../edge_manifest.json    ← seam index map (auto-loaded if present)
```

`resolve_patterns_dir()` calls `product_ingestion.run_manifest.get_latest_panels_dxf_dir()` to find the most recent ingestion output automatically.

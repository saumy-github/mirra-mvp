# Mirra MVP — Step 1 Context Summary
*Last updated: 2026-06-30*

---

## Project Overview

**Mirra** is a virtual try-on platform. Users see how clothes actually look and fit on their own body before buying. The full platform has three sequential steps:

| Step | What it does |
|------|------|
| **Step 1** | Measurement data → personalized 3D digital twin (`.avt` avatar file) |
| **Step 2** | 2D product images → 3D clothing patterns + textures |
| **Step 3** | Avatar + patterns → physics-simulated virtual try-on render |

Step 1 and Step 2 are **independent** and can run in parallel. Step 3 requires both. The digital twin from Step 1 is generated **once per user** and reused for every future try-on session.

**Tech stack**: Python 3.9+, CLO 3D (must be running), MongoDB (measurements collection), REST plugin at `http://localhost:50505`.

**Run command**: `python clo_avatar_generation/run_avatar.py`

---

## Step 1 — 3D Digital Twin Generation

### What It Does

Takes a user's body measurements (entered manually or from MongoDB) and produces a personalized 3D avatar file in T-pose. The avatar is stored once and reused for all future VTO sessions.

- **Input**: Body measurements (height, weight, chest, waist, hip, shoulder, leg) + a base avatar template (`.avt`)
- **Output**: `result_avatar_from_project.avt` — a personalized CLO avatar file
- **Time**: ~30–60 seconds per run 
- **Entry point**: `python clo_avatar_generation/run_avatar.py`

## Measurement Apply Routes

Step 8 supports three routes (auto-selected in priority order):

### 1. `avt_patch` (preferred, default)
The cleanest and most reliable route. Directly writes float values into the binary `.dan` member of the `.avt` zip at known byte offsets. Does not depend on CLO's measurement import API.

**How it works** (`avt_patch.py`):
- `.avt` file = binary header prefix + embedded zip
- Inside the zip is a `.dan` file
- Inside `.dan`, find the `listFeatureValues` marker
- Feature block starts at `marker_offset + len("listFeatureValues") + 273`
- Each feature is a 4-byte little-endian float: `feature_block_start + (feature_index * 4)`
- 57 total feature slots in the block

**Verified working** (run u_001-028): all 6 patched fields read back at exact target values after CLO loads and saves the project. `delta_from_requested = 0.0` for all fields.

### 2. `avatar_properties` (fallback)
Uses CLO's `/avatar/set-properties` API. Requires `has_avatar_property_set` capability from the plugin. Less reliable — `GetAvatarProperties` sometimes doesn't confirm the change.

### 3. `csv` (last resort)
Uses CLO's `/import-avatar-measurements` API with a 2-row CSV (header + values). Seeded from `measurement_template_unconfirmed.csv` (11 columns). Least direct — depends on CLO's internal interpretation.

## Known Hard Blockers / Disabled Code

1. **`/avatars/state` API is disabled** (`step_09_readback.py:14`) — calling CLO's `GetAvatarCount` etc. from the HTTP thread causes an SEH crash on Windows that kills the plugin server. Skipped entirely.

2. **`ExportAVT` is disabled** (`step_11_save_outputs.py:97`) — `ExportAVT` raises SEH exceptions on Windows crashing CLO's main thread. The avatar is instead extracted by unzipping the `.zprj` file.

3. **Error computation is a stub** (`step_10_compute_error.py`) — Step 10 cannot compute actual measurement error because there is no API to read back body measurement values from CLO. All fields are marked `not_read_back`. The binary `verify_avatar_fields` in Step 11 is the only real accuracy check, and it only covers AVT-patched fields.

4. **Weight has no AVT feature index** — `weight_kg` has no verified feature index in the contract. It is applied via CSV only and is never verified after apply.

5. **Male only** — step 3 hard-rejects any gender other than `"male"`. Female fields are reserved in the contract but not active.

## Run History & Apply Route Verdict

28 runs total for user `u_001`. All runs use the same measurements (height 178.5 cm, etc.). The routes were tried in phases as development progressed.

### Full Timeline

| Runs | Route used | Pipeline result | Actual morphing |
|---|---|---|---|
| 001–006 | CSV (implicit) | **Failed** — pipeline collapsed before step 8 | Not applied |
| 007 | `avatar_properties` | Completed (all steps passed) | **Likely NOT applied** — see below |
| 008 | — | Failed before step 8 | Not applied |
| 009–010 | `avatar_properties` | Completed (all steps passed) | **Likely NOT applied** — see below |
| 011 | `csv` | **Failed** at step 8 | Not applied |
| 012–028 | `avt_patch` | **Completed, verified** | ✅ Confirmed exact — delta = 0.0 |

### Route Verdicts

**`csv` — Broken, never worked.**
Tried in runs 001–006 (early) and explicitly in run 011. Failed every time. The CSV template (`measurement_template_unconfirmed.csv`) column names didn't match what CLO expected internally. Run 011 failed at step 8 with the route returning `success: false`.

**`avatar_properties` — Misleadingly "successful", likely ineffective.**
Runs 007, 009, 010 all show pipeline status `"completed"` with step 8 `success: true`. But the confirmation data tells a different story. From run 007's `apply_result.json`:
```
"confirmed_changed_property_count": 0,
"changed_keys": [],
"missing_after_keys": [
    "Across Shoulder (Curvilinear)", "Chest", "Inseam",
    "Low Hip", "Total Height", "Waist"
],
"properties_after": {
    "DivideMesh", "KineticFriction", "SkinOffsetMM", "StaticFriction"
    ← only physics properties, no body measurements
}
```
CLO accepted the API call and returned HTTP 200, but reported `changed=0` and couldn't find any of our body measurement keys after the call. Our own code left a note: *"SetAvatarProperties completed, but GetAvatarProperties did not confirm any requested key change. Visual validation in CLO is still required."*

The pipeline marked step 8 successful because the API returned no error — but the avatar was almost certainly not morphed. There was no binary verification at that time to catch it. This is why the `avatar_properties` route is considered a false positive.

**`avt_patch` — The only route that genuinely works.**
Working since run 012, confirmed with binary readback in every run from that point. The verification in step 11 reads the actual float bytes from the extracted `.avt` and compares them against the requested values. All 6 fields match exactly (`delta_from_requested = 0.0`) in every successful run. This is the ground truth.

### Key Difference: Why avt_patch Actually Works

- **`avatar_properties`**: sends measurement names as string keys to CLO's API → CLO silently ignores keys it doesn't recognise as body measurement properties → avatar unchanged
- **`avt_patch`**: bypasses CLO's API entirely → writes float values directly to binary slots in the `.dan` file → CLO loads an already-morphed file and has no choice but to keep the values

The `avatar_properties` failure revealed that CLO's `SetAvatarProperties` endpoint, at least in this plugin version, does not map our named measurement keys to body shape. It only handles its own internal simulation/physics properties (friction, skin offset, etc.).

## How to Run

**Interactive** (will prompt for user_id, run number, paths):
```bash
python clo_avatar_generation/run_avatar.py
```

**Non-interactive** (automated, uses defaults):
```bash
python clo_avatar_generation/run_avatar.py --non-interactive --user-id u_001
```

**With explicit options**:
```bash
python clo_avatar_generation/run_avatar.py \
  --user-id u_001 \
  --apply-mode avt_patch \
  --measurement-file clo_avatar_generation/input/u_001.measurements.json
```

**Apply modes**: `auto` (default) | `avt_patch` | `avatar_properties` | `csv`

**Check results**: look at `clo_avatar_generation/output/<user_id>-<run_number>/run_summary.json` for step pass/fail, and `measurement_verification.json` for accuracy.

# CLO Workspace — Current State Summary

*Written: 2026-06-30. Describes only what exists right now — no proposed changes.*

---

## What This Folder Is

`clo_workspace/` contains everything needed to build, install, and manage the CLO REST plugin. The plugin is a native library (`.dll` on Windows, `.dylib` on Mac) that CLO 3D loads as a plugin. Once loaded, it runs a local HTTP server on `http://localhost:50505`. Our Python pipeline talks to CLO exclusively through this HTTP server.

## What the Plugin Does

When CLO starts and the user clicks **Plugins → REST Server & Execute**, `DoFunction()` runs on CLO's main thread. It starts the HTTP server on a background thread and registers a timer to automatically drain the command queue. After the first click, everything runs automatically — no further menu clicks needed for normal operation.

The plugin exposes two types of endpoints:

**Write/action endpoints (POST)** — these queue work for CLO's main thread:
- `/new-project` — clears the CLO scene
- `/import-avatar` — loads an OBJ avatar
- `/import-avatar-avt` — loads a native CLO `.avt` avatar template
- `/import-avatar-measurements` — imports measurement CSV
- `/avatar/set-properties` — sets avatar property key-value pairs
- `/import-pattern` — loads a DXF pattern file
- `/arrange-pattern` — places a pattern piece in 3D space around avatar
- `/set-fabric` — assigns a fabric index to a pattern
- `/create-seam` — creates a seam between two pattern edges
- `/simulate` — runs physics simulation for N steps
- `/export` — exports scene to GLB/GLTF
- `/save-project` — saves CLO project as `.zprj`
- `/export-avatar-avt` — exports avatar as `.avt` (see known issues below)
- `/execute` — returns current queue status (does not trigger processing)

**Read/query endpoints (GET)** — return current state:
- `/health` — plugin version, platform, build time
- `/capabilities` — which features are available on this platform
- `/status` — queue size, processing flag, last batch results
- `/debug/import-scales` — scale values used in last avatar/pattern imports
- `/avatar/debug` — avatar import state + arrangement slot info
- `/avatar/native-debug` — native AVT import state
- `/avatar/property-debug` — result of last `set-properties` call
- `/avatars/state` — avatar count, names, properties (see known issues below)
- `/patterns/count` — number of loaded pattern pieces
- `/patterns/{index}` — info about a specific pattern
- `/patterns/{index}/bbox` — bounding box + area of a pattern
- `/patterns/{index}/input` — raw input info for a pattern
- `/patterns/{index}/line-lengths` — line lengths by index (up to 256)
- `/arrangement-list` — CLO's avatar arrangement slots
- `/pattern-arrangements` — which slot each pattern is assigned to
- `/arrangement/debug` — slots + pattern arrangements combined

## Threading Model

Both platforms use the same basic design: the HTTP server runs on a background thread, and CLO's internal API calls must only happen on the main thread. The command queue bridges the two.

```
Python pipeline
    ↓  POST /import-avatar-avt
HTTP thread  →  pushes APICommand to g_commandQueue
    ↓
Timer fires on main thread  →  ProcessCommandQueue() runs
    ↓  calls IMPORT_API->ImportAvatar()  (safe on main thread)
Python polls /status until queue_size == 0
```

### Windows Timer

A Win32 `SetTimer` fires the `QueueDrainTimer` callback every **500ms** via CLO's Win32 message pump. The callback checks if the queue has items and calls `ProcessCommandQueue()` if so.

### Mac Timer

A Qt `QTimer` fires every **200ms** via CLO's Qt event loop. The timer callback runs on the main thread and calls `ProcessCommandQueue()` if there is work.

### How ProcessCommandQueue Works (Both Platforms)

1. Takes the mutex, drains **all** pending commands into a local `batch` vector at once, releases the mutex.
2. Sets `g_queueProcessing = true`.
3. Iterates through the batch and executes each command one by one in order.
4. Sets `g_queueProcessing = false`.

The mutex is only held during the drain step, not during actual CLO API calls.

## Windows vs Mac Differences

### Sync-Read System (Mac Only)

The Mac plugin has a `dispatchSyncRead()` function. When the HTTP thread needs a read result from CLO, it creates a `std::promise`, pushes a command with `isSync=true` to the queue, and blocks (up to 3 seconds) waiting for the main thread to fulfill the promise. Every GET endpoint that needs CLO API data uses this system.

Windows does not have this. On Windows, GET endpoints that need CLO data call CLO APIs directly from the HTTP thread.

### What Works on Mac but Not Windows

| Endpoint | Windows | Mac |
|---|---|---|
| `GET /avatars/state` | Returns hardcoded error | Works via sync-read queue |
| `POST /export-avatar-avt` | Returns failure immediately | Works, calls ExportAVT |
| `GET /capabilities` → `has_avatar_state_readback` | `false` | `true` |
| `GET /capabilities` → `has_avatar_avt_export` | `false` | `true` |

### Exception Handling

Both plugins use `try { ... } catch (const std::exception& e) { ... }` around every HTTP handler and every command in `ProcessCommandQueue`. This catches standard C++ exceptions.

Neither plugin catches Windows SEH (hardware) exceptions via `__try/__except`. This is the root cause of the two disabled Windows endpoints — certain CLO API calls raise SEH exceptions that bypass all C++ catch blocks.

## Known Issues (from v_1.1.1.json)

1. **Native avatar measurement CSV import** — under investigation. Column names in the template CSV do not match what CLO expects (e.g., CLO has "Bust" not "Chest"), causing some measurements to be silently unmatched.

2. **Windows GET read endpoints** — `/patterns/*`, `/arrangement-list`, etc. call CLO APIs directly from the HTTP thread. This works for these specific calls today, but it is the same pattern that caused the `/avatars/state` crash. Mac routes all of these through `dispatchSyncRead` instead.

3. **Windows `/avatars/state`** — CLO's `GetAvatarCount()` and related APIs raise SEH exceptions when called from the HTTP thread on Windows. The endpoint is stubbed: it always returns `{"success": false, "error": "..."}` without calling any CLO API.

4. **Windows `export-avatar-avt`** — CLO's `ExportAVT()` raises SEH exceptions even when called from the main thread on Windows. The queue handler does not call `ExportAVT` at all — it immediately sets `result.success = false` and returns an error message. The workaround used by our Python pipeline is to save a `.zprj` (which is a zip) and extract the `.avt` file from it.

## Build System

### How to Build

```bash
python clo_workspace/build_plugin.py
```

`build_plugin.py` does these steps in order:

1. Reads `.env` for machine config (CLO SDK path, CMake path, etc.)
2. Finds the highest version file in `versions/` (currently `v_1.1.1.json`)
3. Checks version status — refuses to build if status is `"blocked"` (use `--allow-blocked` to override)
4. Generates `shared/PluginBuildInfo.h` with version/platform macros
5. Copies source files from `clo_workspace/windows/` (or `mac/`) and `shared/` into the CLO SDK sample folder at `<CLO_SDK_PATH>/Samples/RestPlugin/`
6. Runs `cmake` configure, then `cmake --build`
7. Copies the built `.dll` or `.dylib` to the vault with a versioned name (`RestPlugin_v1.1.1.dll`)
8. Prints manual install instructions

### What `.env` Needs

Required on all platforms: `CLO_SDK_PATH`, `BUILD_CONFIG`

Windows-specific: `CLO_PLUGINS_DIR`, `CMAKE_GENERATOR` (e.g. `Visual Studio 17 2022`), `CMAKE_ARCH` (x64)

Mac-specific: `CLO_PLUGINS_DIR`, `CMAKE_PREFIX_PATH`, `Qt5_DIR`, `CMAKE_OSX_ARCHITECTURES`

Optional on both: `CLO_PLUGIN_VAULT_DIR` (where versioned builds are stored for rollback), `CMAKE_EXE` (path to cmake if not on PATH)

### Sync-Only Mode

```bash
python clo_workspace/build_plugin.py --sync-only
```

Generates `PluginBuildInfo.h` and copies sources to the SDK folder, but skips the actual CMake build. Useful for updating headers or preparing sources without a full build.

## Vault and Version Management

The vault is a directory outside the repo (set by `CLO_PLUGIN_VAULT_DIR` in `.env`). Every successful build copies the artifact there with a versioned filename. CLO's plugins directory should contain only one versioned plugin at a time.

### Switching Plugin Versions

```bash
# See what's installed and what's in the vault
python clo_workspace/scripts/switch_plugin.py --list

# Activate a specific version from the vault
python clo_workspace/scripts/switch_plugin.py --activate 1.1.0
```

`switch_plugin.py` checks that CLO is not running before copying. It removes any other versioned plugin from CLO's plugins directory after copying the new one.

### Inspecting the Installed Plugin

```bash
python clo_workspace/scripts/get_installed_plugin_info.py
```

Returns JSON showing the active plugin filename, version extracted from filename, file size, and what versions are available in the vault. Exits with code 1 if no plugin artifact is found.

## Shared Header Details

### `CloNativePluginSupport.h`

Defines:
- `NativeAvatarDebugState` struct — holds last AVT import path, success flag, last measurement CSV path, last message
- `BuildNativeAvatarDebugJson()` — builds the JSON response for `/avatar/native-debug` from a `NativeAvatarDebugState` snapshot

Used by both Windows and Mac plugins.

### `PluginBuildInfo.h`

Auto-generated every build. Contains `#define` macros: `MIRRA_PLUGIN_NAME`, `MIRRA_PLUGIN_VERSION`, `MIRRA_PLUGIN_API_VERSION`, `MIRRA_PLUGIN_RELEASE_DATE`, `MIRRA_PLUGIN_RELEASE_STATUS`, `MIRRA_PLUGIN_PLATFORM`, `MIRRA_PLUGIN_PLATFORM_SYNC_STATE`, `MIRRA_PLUGIN_CONTRACT_NAME`, `MIRRA_PLUGIN_CONTRACT_VERSION`, `MIRRA_PLUGIN_BUILD_TIME`. Do not edit by hand.
